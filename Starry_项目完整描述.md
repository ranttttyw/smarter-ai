# Starry · AI 星球漫游 — 完整项目描述

## 一、为什么做这个产品

### 核心痛点洞察

作为一个日常高频使用多个 AI 模型的用户，我发现了一个普遍存在但被忽视的问题：**用户在面对越来越多的 LLM 选择时，缺乏直觉化的比较工具和决策依据。**

具体表现在三个层面：

**1. 选择困难**：市面上有 ChatGPT、Gemini、DeepSeek、GLM 等十几个模型，每个都说自己最好，但用户不知道"对我的问题，哪个最合适"。现有的 benchmark 是面向开发者的，普通用户看不懂 MMLU 分数。

**2. 比较低效**：想对比只能手动复制粘贴同一个问题到多个平台，然后来回切换窗口对比回答——费时、割裂、无法形成记忆。

**3. 缺乏自我认知**：用户的模型偏好其实反映了他们的思维方式和价值观（比如有人偏好简洁直接的回答，有人喜欢详细论证），但没有任何工具帮助用户认识到"我是什么样的 AI 使用者"。

### 产品定位

Starry 不是又一个"AI 聚合器"，而是一个 **AI 宇宙探索体验**——用"星球探索"的隐喻把枯燥的模型对比变成一场沉浸式的发现之旅。每个 AI 模型是一颗星球，用户的每次选择都在绘制自己的"星际人格画像"。

### 目标用户

中国年轻知识工作者（学生、内容创作者、分析师、PM），日常使用 AI 但不是技术极客，更关注"哪个回答更合我意"而不是"哪个模型跑分更高"。

## 二、技术架构与技术栈

### 整体架构

```
┌─────────────────────────────────────────────────┐
│                  Frontend                        │
│         Vanilla JS + Canvas + CSS 3D             │
│              (Single Page App)                   │
└──────────────────┬──────────────────────────────┘
                   │ HTTP (POST /api/compare, /api/chat)
┌──────────────────▼──────────────────────────────┐
│              Backend (Express.js)                │
│                Port 3001                         │
├─────────┬─────────┬──────────┬──────────────────┤
│ Gemini  │DeepSeek │   GLM    │  Meta-Analysis   │
│ Client  │ Client  │  Client  │   (DeepSeek)     │
└────┬────┴────┬────┴────┬─────┴────────┬─────────┘
     │         │         │              │
  Google    DeepSeek   Zhipu AI     DeepSeek
  Gemini    API        API          API (分析层)
  2.0 Flash            GLM-4-Flash
```

### 技术栈详情

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **前端渲染** | Vanilla HTML/CSS/JS | 零依赖、快速迭代、完全控制动画细节 |
| **星空系统** | Canvas 2D + requestAnimationFrame | 高性能粒子渲染，250+星体 60fps |
| **星球视觉** | CSS 3D (radial-gradient + box-shadow + pseudo-elements) | 无需 WebGL 即可实现 3D 球体效果 |
| **后端框架** | Node.js + Express | 轻量、适合 I/O 密集型的多 API 并发调用 |
| **并发请求** | Promise.allSettled | 多模型并行查询，单个失败不影响整体 |
| **元分析引擎** | DeepSeek API（自建分析 prompt） | 用 AI 分析 AI，生成跨模型维度评分 |
| **数据持久化** | localStorage | MVP 阶段轻量存储用户偏好数据 |
| **API 认证** | JWT (HS256) / Bearer Token | GLM 需要 JWT 签名，其他用标准 Bearer |

### 核心 API 设计

**POST /api/compare** — 多模型并行对比
```json
// Request
{ "prompt": "用户问题", "models": ["gemini", "deepseek", "glm"] }

// Response
{
  "results": [
    { "modelId": "gemini", "content": "Gemini的回答...", "error": null },
    { "modelId": "deepseek", "content": "DeepSeek的回答...", "error": null },
    { "modelId": "glm", "content": "GLM的回答...", "error": null }
  ],
  "analysis": {
    "summary": "三个模型在语气和结构上差异明显...",
    "chips": ["DeepSeek最委婉", "Gemini最简洁"],
    "dimensions": ["切题度", "专业度", "礼貌感", "简洁度"],
    "models": {
      "gemini": {
        "tags": [{"label": "简洁", "type": "quality"}],
        "stats": {"切题度": 92, "专业度": 85, "礼貌感": 70, "简洁度": 95},
        "keyPoint": "直接说明理由，格式规范但略显冷淡"
      }
    }
  }
}
```

**POST /api/chat** — 单模型深度对话
```json
// Request
{ "model": "deepseek", "messages": [{"role": "user", "content": "..."}] }
// Response
{ "content": "模型回答..." }
```

## 三、迭代过程——每一步为什么这么做

### 迭代 0：需求分析与产品定义

**做了什么**：
- 明确核心用户场景：写作辅助（小红书文案、邮件）+ 思辨讨论（哲学问题、人生选择）
- 选定 3 个模型（Gemini Flash、DeepSeek V3、GLM-4-Flash）：覆盖国际/国内、不同风格
- 设计"AI 宇宙"核心隐喻，把模型选择变成星球探索

**为什么**：
- 3 个模型是 MVP 的最佳数量——少于 3 无法体现对比价值，多于 3 会让界面拥挤
- 选择免费/低价 API 保证可持续运行
- "宇宙"隐喻解决了"工具感太强"的问题，让对比过程有情感价值

---

### 迭代 1：后端多模型并发架构

**做了什么**：
- 搭建 Express.js 服务器，封装 5 个 LLM API Client
- 使用 `Promise.allSettled` 实现并行查询（而非串行），3 个模型同时发送请求
- 特殊处理 GLM 的 JWT 认证：自动检测 API Key 格式（id.secret vs 纯 token），动态生成 HS256 签名

**为什么**：
- 串行调用 3 个模型需要 6-9 秒，并行只需最慢那个模型的时间（约 2-3 秒）
- `allSettled` 而非 `all`：即使某个模型超时或报错，其他结果仍然返回，不影响用户体验
- GLM 的认证机制和其他模型不同，需要额外的 JWT 签名逻辑

**结果**：响应时间从预估的 9 秒降至 2-3 秒，单模型故障不影响整体。

---

### 迭代 2：DeepSeek Meta-Analysis 分析引擎

**做了什么**：
- 设计了一个"用 AI 分析 AI"的元分析层：把所有模型的回答送给 DeepSeek，让它从多个维度评估差异
- 根据问题类型动态选择评估维度（4 维）：
  - 写作类 → 切题度 / 种草感 / 情绪感染 / 口语化
  - 邮件类 → 切题度 / 专业度 / 礼貌感 / 简洁度
  - 产品介绍 → 切题度 / 说服力 / 差异化 / 可信度
  - 哲学思辨 → 切题度 / 深度 / 共鸣 / 创新性
- 每个模型生成：tags（自动标签）、stats（维度分数 0-100）、keyPoint（一句话精华）

**为什么**：
- 直接把 3 段回答摆在用户面前是"信息过载"——用户需要的是"告诉我哪里不同"
- 静态维度不适用所有场景（写邮件和写文案的评价标准完全不同）
- 分数 + 标签 + 一句话总结，三层信息密度满足"快速扫一眼"到"仔细对比"的不同需求

**结果**：用户不需要逐字阅读 3 段回答，看一眼维度分数和标签就能做出选择。

---

### 迭代 3：Canvas 粒子星空系统

**做了什么**：
- 用 Canvas 2D API 实现全屏星空背景：250 颗星体，带闪烁（twinkle）和漂移（drift）效果
- 每 4-8 秒随机生成流星动画
- 使用 `requestAnimationFrame` 保证 60fps 平滑渲染
- 星体有不同亮度和大小，模拟真实星空的层次感

**为什么**：
- 静态背景图无法传递"宇宙在运动"的感觉
- Canvas 粒子系统性能开销小（250 个点的绘制每帧 <1ms），不影响页面交互
- 流星增加"惊喜感"，让用户在等待响应时不觉得无聊

**结果**：页面从"白色工具界面"变成"活的宇宙空间"，用户反馈"第一眼就觉得不一样"。

---

### 迭代 4：CSS 3D 星球系统

**做了什么**：
- 为每个模型设计独特的 3D 星球外观（纯 CSS，无 WebGL）：
  - **Gemini**（蓝色水晶球）：双层 radial-gradient + 小月亮轨道动画
  - **DeepSeek**（紫色星云球）：内发光 + 土星环（conic-gradient + 旋转动画）
  - **GLM**（青色极光球）：极光 shimmer 动画 + 内部流动效果
- 交互状态系统：
  - 默认：浮动动画（float1/2/3，不同节奏避免同步）
  - 悬停：放大 1.12x + 光晕增强
  - 选中：脉冲光圈 + 亮度提升 30%
  - 思考中：旋转动画（planetSpin）
  - 未选中：变暗（opacity 0.35）+ 缩小

**为什么**：
- 第一版星球"太丑了"（用户原话），只是简单的圆形色块
- 每个模型需要视觉上的"性格"——Gemini 清冷理性、DeepSeek 神秘深邃、GLM 科技未来
- 交互状态让用户知道"系统在干嘛"：旋转 = 思考中，变暗 = 不是你选的

**结果**：星球从"色块"变成有辨识度的视觉符号，用户开始用"那个紫色的"代指 DeepSeek。

---

### 迭代 5：结果展示与交互优化

**做了什么**：
- 动态网格布局：根据返回结果数量自动切换 1/2/3 列（显式 CSS class，不依赖 auto-fit）
- 问题气泡："你问了：xxx" 显示在结果上方
- 骨架屏 loading：思考时直接在结果区显示骨架卡片，而非在输入框里显示文字
- 回答后星球隐藏上移（`.hidden-up`），结果占满全屏

**为什么**：
- `repeat(auto-fit, minmax(300px, 1fr))` 在只有 2 个结果时不会填满屏幕宽度——这是 CSS Grid 的已知行为
- 用户反馈"Loading 文字还留在对话框里很奇怪"
- 问完问题后星球不消失会和结果卡片抢空间

**结果**：修复了 5 个 UI 层面的 bug，体验从"能用"变成"流畅"。

---

### 迭代 6："说人话"评价系统

**做了什么**：
- 6 个 emoji 评价气泡（替代传统的 1-5 星评分）：
  - 🤯 格局打开了
  - 💡 说到心坎里了
  - 😂 笑死我了
  - 🙃 太官方了吧
  - 🫠 有点啰嗦
  - 🤖 没感情的AI
- 评价后保留"进入深聊"按钮（而非整个面板消失）
- 评价数据通过 localStorage 持久化，key: `starry_reactions`

**为什么**：
- 星级评分太"工具化"，和宇宙探索的浪漫感冲突
- 用户说"减弱工具性"——所以用日常对话中的口头表达代替数值评分
- 第一版评价后 deep chat 按钮消失了（reactAndClose 删除了整个面板），需要拆分函数

**结果**：评价从"打分任务"变成"表达感受"，和产品调性一致。

---

### 迭代 7：星际人格画像系统

**做了什么**：
- 六维雷达图（Canvas 绘制）：深度思考 / 实用主义 / 创意追求 / 情感共鸣 / 探索精神 / 审美偏好
- 基于用户的所有选择和评价数据，自动计算六维得分
- 顶部 2 个最高维度组合生成人格标签：
  - 深度思考 + 创意追求 → 浪漫哲学家
  - 实用主义 + 深度思考 → 优雅实干家
  - 探索精神 + 审美偏好 → 星际探险家
  - ……（多种组合）
- 侧边栏"星际护照"展示：人格标签 + 雷达图 + 使用统计

**为什么**：
- 用户的模型选择模式暗含了他们的价值倾向——选 DeepSeek 的人可能更追求深度，选 Gemini 的人可能更看重效率
- 雷达图是最直觉的多维数据可视化方式
- "人格画像"把被动的工具使用变成主动的自我认知，增加用户粘性和分享动力

**结果**：产品从"比较工具"升级为"AI 使用者的人格测试"，增加了社交传播潜力。

**做了什么**：

  1. **PRD**：问题定义、用户故事、功能规格、验收标准

- 文档体系证明了"我不是只会写代码的开发者，我是从用户洞察出发的产品人"

**结果**：5 份专业 PM 文档，可直接用于面试作品集展示。

## 四、关键产品决策与思考

### 为什么用"宇宙"隐喻？
AI 模型对比本质上是"探索未知"——你不知道哪个模型最适合你，就像不知道哪颗星球适合居住。这个隐喻把功能性需求（比较模型）和情感需求（探索的乐趣）结合在一起。

### 为什么让 DeepSeek 分析其他模型？
这是一个有趣的"AI 评价 AI"设计。选择 DeepSeek 作为分析引擎是因为它的中文理解能力强且 API 成本低。每次分析只传前 600 字符控制 token 消耗。

### 为什么不用 React/Vue 而用 Vanilla JS？
MVP 阶段速度最重要。单文件 2400 行虽然不是最佳工程实践，但让我可以在几小时内完成从想法到可交互原型的全流程。如果产品验证成功，重构为组件化框架是 roadmap 里的下一步。

### 为什么北极星指标是 WAU 而不是 DAU？
AI 对比不是日常行为——用户不会每天比较模型。但一周内遇到一个需要认真选择 AI 工具的场景是合理的。WAU 更真实地反映产品价值。
